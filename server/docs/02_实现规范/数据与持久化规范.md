# 数据与持久化规范

## 当前持久化基线

- 数据库：`SQLite`
- ORM：`Prisma`
- 默认连接：`server/.env` 中的 `DATABASE_URL`
- 当前业务模型：`Player`、`PlayerProgress`、`PlayerSkillProgress`、`PlayerBattleSession`

## Player 模型

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `String` | 主键，默认 `cuid()` |
| `account` | `String` | 唯一账号 |
| `nickname` | `String` | 当前默认与账号一致 |
| `passwordHash` | `String` | 密码的 SHA-256 哈希值 |
| `lastLoginAt` | `DateTime?` | 最近登录时间 |
| `createdAt` | `DateTime` | 创建时间 |
| `updatedAt` | `DateTime` | 更新时间 |

## PlayerProgress 模型

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `playerId` | `String` | 与 `Player` 一对一 |
| `currentChapterId` | `Int` | 当前章节 |
| `highestUnlockedChapterId` | `Int` | 已解锁的最高章节 |

## PlayerSkillProgress 模型

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `playerId` | `String` | 所属玩家 |
| `skillId` | `Int` | 对应 `Skill` 配置表中的功法 ID |
| `level` | `Int` | 当前功法等级 |
| `exp` | `Int` | 当前功法经验 |
| `stars` | `Int` | 当前功法星级 |

## PlayerBattleSession 模型

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `playerId` | `String` | 所属玩家，当前设计为一人仅保留一个活动/最近会话 |
| `chapterId` | `Int` | 当前战斗所属章节 |
| `layerIndex` | `Int` | 当前层 |
| `waveIndex` | `Int` | 当前波次 |
| `missionId` | `Int?` | 当前使用的 Mission 配置 ID |
| `enemyId` / `enemyName` | `Int?` / `String?` | 当前敌人摘要 |
| `status` | `String` | 当前会话状态，如 `active`、`settled` |
| `result` | `String?` | 结算结果，如 `victory`、`defeat` |

## 当前实现方式

- Prisma schema 定义在 `server/prisma/schema.prisma`。
- 服务启动时会执行 `ensureDatabaseSchema()`，保证 `Player` 表存在。
- 账号注册、登录逻辑集中在 `server/src/services/authService.js`。
- 主页、章节、功法和战斗预览相关数据由服务端配置表与数据库共同提供。
- 战斗占位会话会真实写入 `PlayerBattleSession`，不再允许前端自行拼装战斗业务数据。

## 开发约束

- 新增业务表时，优先通过 Prisma schema 明确模型，再决定是否保留启动时的补表逻辑。
- 对外返回玩家信息时，必须使用脱敏后的结构，不能把 `passwordHash` 暴露给接口层。
- 账号相关规则变更时，需要同时检查数据库字段、service 逻辑和接口文档是否一致。
- 只有账号和服务端业务数据允许落库；前端不应承担游戏数据持久化职责。
- 功法、战斗、章节推进等动态业务状态必须由服务端写库或由服务端配置生成，前端只保留账号体验相关本地缓存。
