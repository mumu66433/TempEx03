# 服务端目录结构

## 当前结构

```txt
server/
  prisma/
    schema.prisma
  src/
    app.js
    index.js
    server.js
    assets/
      config/
    config/
    routes/
    services/
  docs/
```

## 目录职责

| 路径 | 职责 |
| --- | --- |
| `server/src/index.js` | 服务启动入口，仅负责调用启动流程 |
| `server/src/server.js` | 启动服务、初始化数据库、预加载 Excel 配置 |
| `server/src/app.js` | 组装 Express 中间件与 API 路由 |
| `server/src/routes` | 定义 HTTP 路由和请求响应边界 |
| `server/src/services` | 封装业务逻辑、数据库调用与配置聚合 |
| `server/src/config` | 环境变量、Prisma 客户端、Excel 加载器等基础设施 |
| `server/src/assets/config` | Excel 配置表原始文件 |
| `server/prisma` | Prisma schema 与本地 SQLite 数据文件 |
| `server/docs` | 程序代理维护的后端文档 |

## 后续扩展建议

- 新增路由时，优先按业务主题拆分文件，不把所有接口堆在单个路由文件中。
- 新增业务逻辑时，优先在 `services` 内新增专题 service，再由 route 调用，不把复杂逻辑直接写进路由层。
- 如果后续出现 DTO、校验器、常量、测试等新类型文件，建议在 `src` 下新增专门目录，而不是混入 `services`。
- 若引入后台管理、定时任务或脚本任务，建议在 `server/src` 下新增单独入口，避免与 API 启动流程耦合。

## 当前边界

- 当前服务端仍是轻量模板，目录设计优先简单清晰，不提前引入过重分层。
- 只有当业务模块明显增多时，再考虑增加 controller、repository 等更细层级。
