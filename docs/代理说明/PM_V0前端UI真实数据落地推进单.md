# [PM] V0 前端 UI 真实数据落地推进单

## 一、推进目标

本推进单用于把 `V0_原型验证` 的 UI 原型、真实后端数据、前端 Phaser 场景实现和 PM 验收口径统一起来。

目标不是让前端只做静态还原，而是形成可运行闭环：

- 页面结构、视觉层级、状态表现符合 `docs/原型制作` 下的 V0 UI 原型和标注。
- 章节、功法、战斗、奖励、进度等玩法数据全部来自服务端接口或服务端配置。
- 前端只保存账号相关本地信息，不保存玩法数据。
- 每个页面至少覆盖加载中、请求失败、空数据、禁用、未开放等必要接口态。
- PM 可以按页面和流程逐项验收，不再依赖口头理解。

## 二、PM 暂定范围结论

以下结论先作为本轮推进默认依据；如用户后续明确调整，以用户最新结论为准。

| 待确认项 | 暂定结论 | 处理理由 |
| --- | --- | --- |
| 游戏入口页 | V0 保留 | 现有前端已有登录成功后的 `LoginScene`，且 UI 文档明确入口页属于 V0 范围。 |
| 登录/注册表单 | 不重做 | 继续沿用现有账号登录层，符合 README 与 UI 文档说明。 |
| 三选一 | 作为 V0 核心目标推进真实接入 | 这是 V0 主循环的一部分，但需要先补后端契约，再由前端实现。 |
| 功法升级 | V0 先展示状态，不接真实升级操作 | 当前缺少明确升级接口，先避免前端自行伪造行为。 |
| 调试入口 | 开发/原型环境放章节页右上角 | 与现有 UI 补充说明一致，正式演示环境应隐藏或弱化。 |
| 第一轮闭环 | 先保住登录 -> 入口 -> 章节 -> 战斗 -> 结算 -> 章节刷新 | 该闭环已有代码基础，适合作为真数据联调基线。 |

## 三、阶段拆分

### 阶段 1：契约定稿

目标：让 UI、后端、前端不再各自猜字段、状态和交互。

| 角色 | 负责目录 | 交付物 |
| --- | --- | --- |
| UI 设计代理 | `docs/原型制作` | 将 `V0_前端UI对接单_示例.md` 升级为正式对接单，补齐页面状态、交互、文案边界、资源说明和验收口径。 |
| 后端代理 | `server`、`server/docs` | 输出 V0 前端所需接口契约，明确章节、首页、功法、战斗、结算、三选一的数据字段和样例。 |
| 前端代理 | `client` | 根据正式对接单和接口契约整理前端 ViewModel 清单，并新增当前实现与 UI 设计稿一致性自查报告，标出当前已接入、设计差异和缺口。 |
| PM | `docs/代理说明` | 维护本推进单，跟踪阻塞项，审核各角色是否越权和是否完成推送。 |

阶段 1 完成标准：

- UI 有正式对接单，不再只有示例。
- 后端有接口字段说明和真实样例。
- 前端明确哪些页面能直接实现，哪些等待后端或 UI 补充，并提交当前实现与 UI 设计稿一致性自查报告。

### 阶段 2：真数据最小闭环

目标：优先打通现有基础最好的主路径。

闭环路径：

```txt
账号登录 / 自动登录 -> 游戏入口页 -> 章节页 -> 开始战斗 -> 战斗页文字战报 -> 结算页 -> 返回章节页并刷新进度
```

验收重点：

- 章节页不展示伪造章节，失败和空数据有明确提示。
- 战斗页的 HUD、战报、血条、日志、摘要来自服务端返回。
- 结算页展示真实胜负、章节推进结果和战报摘要。
- 返回章节后，当前章节和最高解锁章节刷新正确。

### 阶段 3：V0 完整主循环

目标：补齐功法三选一和功法页，使流程更贴近完整 V0 设计。

需要后端优先确认：

- 三选一候选功法接口。
- 确认选择功法接口。
- 刷新一次候选功法接口或规则。
- 本局构筑条数据来源。
- 满星、不可选、随机池收束等状态字段。

需要前端实现：

- 三卡同屏弹窗。
- 未选择、已选择、刷新 loading、确认提交中、不可选卡等状态。
- 确认选择后回到战斗流程。
- 功法列表页统一 V0 UI 语言。
- 功法详情浮层展示真实字段，升级按钮 V0 仅展示或提示暂未开放。

## 四、角色派单草案

### UI 设计代理

日志前缀：`[UI]`

维护目录：`docs/原型制作`

任务：

- 新增正式版 `V0_前端UI对接单.md`。
- 将现有示例中的格式转为正式开发依据。
- 每页至少覆盖默认、加载、失败、空数据、禁用、选中、提交中状态。
- 明确哪些视觉元素必须按 SVG/标注还原，哪些可由代码绘制，哪些允许占位。
- 对功法三选一、功法详情、结算页补足真实数据下的长文案和空值表现规则。

禁止：

- 不修改 `client`、`server`。
- 不自行改变策划规则或接口字段。

### 后端代理

日志前缀：`[后端]`

维护目录：`server`、`server/docs`

任务：

- 新增或更新 V0 API 契约文档。
- 明确当前已有接口的返回字段、空值规则和样例。
- 评估并补齐三选一所需接口。
- 明确功法升级在 V0 是否仅展示；如不实现，接口文档中需写明。
- 保证前端页面所需数据来自服务端，不要求前端硬编码玩法数据。

禁止：

- 不修改 UI 原型和前端页面实现。
- 不把数据库运行产物混入提交，除非明确说明需要提交。

### 前端代理

日志前缀：`[前端]`

维护目录：`client`

任务：

- 先整理 V0 页面 ViewModel 清单。
- 新增 `client/docs/V0_UI一致性自查报告.md`，对照 UI 设计稿检查当前已实现界面是否一致。
- 自查范围至少包含 `LoginScene`、`HomeScene`、`BattleScene`、`SkillScene`、`ResultScene`、`v0ui.js` 与旧 `ui.js` 使用情况。
- 自查报告需逐页列出设计稿来源、当前实现文件、已一致项、差异项、风险等级和建议修复顺序。
- 特别指出功法页是否仍偏旧 UI 风格，三选一是否已实现，以及缺哪些后端/API/前端内容。
- 章节页、入口页、战斗页、结算页优先按真数据闭环完善。
- 功法页统一到 V0 UI 组件和视觉语言。
- 等后端三选一契约明确后，实现三选一弹窗和本局构筑条。
- 所有接口态必须明确，不用假数据掩盖失败。

禁止：

- 不修改后端接口和数据库逻辑。
- 不直接改写 UI 对接单。

## 七、当前风险

- `server/prisma/dev.db` 当前已有未提交变更，后续提交必须避免误带入，除非对应代理明确说明这是必要数据库变更。
- 三选一当前缺少清晰接口契约，前端如果先做只能做临时表现，真实主循环会返工。
- 功法页当前实现风格与 V0 UI 组件不完全一致，需要前端统一。
- 前端一致性自查属于本轮新增任务，PM 后续应优先审核报告中列出的高风险差异。
- 战斗服务当前偏整场模拟；如要严格实现“普通层胜利 -> 三选一 -> 继续下一层”，需要后端拆更细的战斗阶段或提供阶段结果。
- `V0_前端UI对接单_示例.md` 仍是示例文档，不能长期作为正式验收依据。
