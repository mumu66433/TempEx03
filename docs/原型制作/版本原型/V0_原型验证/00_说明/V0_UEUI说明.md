# V0 UE / UI 说明

本目录是 `V0_原型验证` 的设计稿和交互说明目录，直接服务当前版本开发。

## V0 设计目标

- 验证“章节推进 + 自动战斗 + 功法三选一”是否成立
- 战斗表现先采用“文字战斗主表现区”临时方案，优先验证章节推进与自动战斗主循环
- 让玩家在 10 分钟内理解主循环
- 保证页面少、层级浅、操作路径清晰

## V0 设计范围

- 游戏入口页
- 章节页
- 战斗页
- 战斗状态稿：普通波 / 精英波 / BOSS层 / 受击 / 死亡 / 波次切换 / 暂停
- 功法三选一弹窗
- 功法列表页
- 功法详情浮层
- 功法状态稿：未解锁 / 可升级 / 满星 / 品质筛选
- 通关结算页
- 失败结算页

## V0 不做

- 多角色选择页
- 装备页
- 法宝页
- 帮派与社交页
- 活动页和商城复杂页

## 目录说明

- `01_流程与结构`：信息架构、页面流转、交互路径
- `02_界面设计`：V0 页面设计稿和状态稿
- `03_切图与标注`：尺寸标注、交互标注、研发落地提示

## 研发实现边界

- README 约定游戏主界面由 `Phaser 3` 场景渲染；V0 UI 默认按 `750 x 1624` 设计，再由客户端等比缩放。
- `V0_01_登录界面.svg` 实际为“账号登录成功后的游戏入口页”，不是账号密码输入页；账号登录 / 注册表单仍沿用现有登录层，不在本目录重复设计账号表单。
- 默认主路径为：账号登录 / 注册层 -> 游戏入口页 -> 章节页；如程序为了降低实现成本，临时改为“登录成功后直接进入章节页”，需在联调记录中注明，并保留入口页资源供后续回补。
- `V0` 战斗页本轮临时调整为“文字战斗表现”，中心区不再绘制人物 / 敌人图形，仅保留文字战报、回合日志、波次切换和战斗摘要。
- 顶部 HUD、底部构筑条、暂停面板、三选一弹窗、结算页可使用 DOM 覆盖层或 Phaser UI 绘制；无论采用哪种实现，均以本目录的层级、状态和标注文档为准。
- 战斗页必须保留“顶部 HUD + 中心主表现区 + 底部构筑条”三段式锚点；中心区虽然改成文字战报，但仍是主表现区，不能退化为开发调试面板式平铺说明。
- 前端只保存账号相关信息，章节、功法、战斗、奖励和角色属性均以服务端配置和接口返回为准。
- 所有依赖服务端数据的页面，至少要预留“加载中”“请求失败”“空数据 / 无有效配置”三类接口态；原型阶段可以只做轻量提示，不必为每类接口态单独新增完整视觉页。

## V0 文字战斗临时方案

- 本版 `V0` 以验证章节推进、自动战斗主循环、结算回流是否顺畅为第一目标，因此战斗中心区暂不制作角色、敌人和技能动画演出。
- 战斗页中心主表现区固定采用四个模块组织信息：
  - `最新战报`：放大展示当前关键事件，例如击中、暴击、格挡、击败。
  - `回合日志`：按时间顺序滚动显示最近若干条战斗记录，默认自动滚动到最新内容。
  - `波次切换横幅`：用于承接“第 N 波完成 / 下一波来袭 / BOSS 出现”等关键节点。
  - `战斗摘要`：显示累计击败、累计受击、当前主功法、战斗耗时等局内摘要。
- 顶部 HUD 继续承载章节、层波、我方 / 敌方状态；底部继续承载本局构筑条和暂停按钮，供前端保持原型三段式骨架实现。
- 通关 / 失败结算页需继续承接本局文字战报摘要，形成从战斗页到结算页的一致信息链路。

## 当前临时约定

- 暂停面板点击“返回章节”时，默认视为放弃本局，并弹出二次确认；确认后退出本局并回到章节页，取消则留在暂停面板。
- 底部未开放页签 `角色`、`商城` 在 V0 只保留占位，不触发页面跳转；点击时仅给轻提示“暂未开放”。
- 战斗页底部构筑条在功法数量超过单屏展示数量时，默认使用横向滚动查看，不新增二级分页。
- 战斗页中心文字战报区默认保留最近 5-7 条日志的可视高度，超出后继续向下滚动，不新增独立日志二级页。

## 待确认项

- 是否保留“游戏入口页”作为正式主流程的一环，还是在 V0 期间直接落到章节页。
- 功法详情中的“升级”按钮在 V0 是否接真实升级能力；当前文档先按“可显示升级状态，但是否开放实际升级行为待策划确认”执行。
