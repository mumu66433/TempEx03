# 项目代理说明

## 一、全局原则

- 在进行任何修改之前，必须先阅读 [README.md](./README.md)，并将其作为本项目的首要规范。
- 如果任务要求与 `README.md` 冲突，默认以 `README.md` 为准，除非用户明确要求覆盖。
- 实现时请保持和仓库中描述的架构、启动流程、数据规则一致。
- 除非用户明确要求，否则不要在 `main` 分支上直接开发、提交或推送。

## 二、角色与目录映射

- 当前项目默认在 `dev` 单分支下协作。
- PM / 审核管理代理：负责项目文档、协作调度、审核记录和整体推进。
- 策划设计代理：主要维护 `docs/策划方案`。
- UI 设计代理：主要维护 `docs/原型制作`。
- 程序制作代理：主要维护 `client`、`server`、根目录工程脚本和工程配置。

## 三、启动检查规则

- 每次开始工作前，必须检查当前 Git 分支，确认当前仍在项目统一使用的 `dev` 分支。
- 默认以用户明确指定的身份、当前任务类型和准备维护的目录共同确认自身角色，并读取 `docs/代理说明` 下对应的角色说明文件。
- 如果当前任务无法匹配任何角色，应先暂停并询问用户，不要直接修改文件。
- 如果当前任务与当前角色职责不一致，应先暂停并询问用户是否调整分工。
- 如果是在清理上下文后的新对话，必须先读取自身角色对应的状态文件，再继续工作。

## 六、目录权限边界

- 策划设计代理主要维护 `docs/策划方案`，其他目录默认只读。
- UI 设计代理主要维护 `docs/原型制作`，其他目录默认只读。
- 程序制作代理主要维护 `client`、`server`、根目录工程脚本和工程配置；`docs/策划方案` 与 `docs/原型制作` 默认只读。
- PM / 审核管理代理主要负责统一任务分配、审核记录、项目文档和协作协调。
- 非本角色维护目录默认只读，如需跨角色修改，必须先说明原因并征得用户同意。
- 多代理协作默认通过各自目录分工完成，不再通过独立工作分支拆分任务。
- 如需提交，代理只能提交自己负责目录和自己工作目录内的改动，不得顺带提交其他代理目录中的未授权变更。

## 七、日志与标识规则

- 代理输出进度日志、说明日志、协作记录时，必须带上可读前缀，明确标识消息来自哪个代理。
- 日志前缀建议使用统一格式：`[PM]`、`[策划]`、`[UI]`、`[程序]`。
- 多代理并行协作时，所有阶段性同步、问题上报、完成回报都应保留此前缀，避免混淆来源。
- 主代理在创建子代理时，必须在分派信息中明确写出该子代理的角色、可维护目录、禁止触碰目录和日志前缀。
- 子代理在开始工作前，必须先复述并遵守自己被分派的角色与目录边界；若分派信息不清晰，必须先回报主代理确认。

## 八、提交与推送规范

- 每次完成修改后，必须提交代码，并使用中文提交信息说明本次修改内容。
- 当前项目采用单分支协作，所有已完成且确认可提交的改动都应及时提交并及时推送到 `origin/dev`。
- 如需提交，只提交当前代理负责目录与自身工作目录内的改动，不得混入其他代理的改动。
- 不再将“为不同 sub agent 建立独立工作分支”作为默认协作方式。
- 由于当前不采用多分支并行开发，默认不存在分支间合并流程；PM / 审核管理代理主要负责审核提交内容和协调修改顺序。

## 九、上下文清理与协作状态恢复

- 当用户要求某个代理清理上下文、重置对话或开启新对话前，该代理必须先更新自己的角色状态文件。
- 角色状态文件至少记录：当前角色、当前工作树路径、当前分支、远端分支、正在进行的任务、已完成但未审核的目录成果或提交、已派生子代理、需要用户确认的问题、最近一次提交状态。
- 如果当前代理管理了子代理，必须先要求所有子代理各自更新自己的角色状态文件；确认子代理状态已记录后，当前代理再更新自己的角色状态文件。
- 清理上下文后的新对话中，代理必须先读取 `README.md`、`AGENTS.md`、自身角色说明文件和自身角色状态文件，再继续工作。
- 如果角色状态文件缺失、过期，或与 Git 实际状态冲突，必须先通过 `git status`、`git branch -a -vv`、`git worktree list`、`git log` 等命令核对，并向用户说明恢复结果。
- 主代理在重新开始工作后，必须先判断当前是否已经有对应的 subagent；如果没有，且当前任务涉及多个角色分工，就必须根据本文件的目录分工创建对应的 subagent，再继续处理任务。
- 主代理创建 subagent 时，必须先确认子 agent 是否已经存在；如果不存在，就先创建对应子 agent，并为其命名后再继续工作。
- 子 agent 在准备提交前，必须先确认本地 `dev` 已同步到当前工作区，再基于同步后的状态提交自己的改动，避免覆盖其他代理刚提交的内容。
- 子 agent 如需同步 `dev`，推荐按以下顺序执行：
  - 先 `git fetch origin dev`
  - 再 `git pull --ff-only origin dev`
  - 如有本地改动，先确认只涉及自身负责目录，再处理同步冲突
  - 确认当前提交只覆盖自身负责目录后，再继续提交和推送
- 主代理需要每 10 分钟检查一次是否有子代理已完成但未提交或未推送的目录成果；如果有，就优先安排提交、推送和状态更新。
- 在完成协作状态记录之前，不要主动清理上下文，也不要要求子代理清理上下文。
