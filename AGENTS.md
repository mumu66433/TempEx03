# 项目代理说明

## 一、全局原则

- 在进行任何修改之前，必须先阅读 [README.md](./README.md)，并将其作为本项目的首要规范。
- 如果任务要求与 `README.md` 冲突，默认以 `README.md` 为准，除非用户明确要求覆盖。
- 实现时请保持和仓库中描述的架构、启动流程、数据规则一致。
- 除非用户明确要求，否则不要在 `main` 分支上直接开发、提交或推送。

## 二、分支与角色映射

- `dev`：默认开发与审核管理分支。
- `design/planning`：策划设计代理。
- `design/ui`：UI 设计代理。
- `feature/program`：程序制作代理。

## 三、启动检查规则

- 每次开始工作前，必须检查当前 Git 分支。
- 根据当前分支确认自身角色，并读取 `docs/代理说明` 下对应的角色说明文件。
- 如果当前分支无法匹配任何角色，应先暂停并询问用户，不要直接修改文件。
- 如果当前任务与当前角色职责不一致，应先暂停并询问用户是否切换分支或调整任务。
- 如果是在清理上下文后的新对话，必须先读取自身角色对应的状态文件，再继续工作。

## 六、目录权限边界

- 策划设计代理主要维护 `docs/策划方案`，其他目录默认只读。
- UI 设计代理主要维护 `docs/原型制作`，其他目录默认只读。
- 程序制作代理主要维护 `client`、`server`、根目录工程脚本和工程配置；`docs/策划方案` 与 `docs/原型制作` 默认只读。
- PM / 审核管理代理主要负责 `dev` 分支整合、审核记录、项目文档和分支协调。
- 非本角色维护目录默认只读，如需跨角色修改，必须先说明原因并征得用户同意。

## 七、提交与推送规范

- 每次完成修改后，必须提交代码，并使用中文提交信息说明本次修改内容。
- 每次提交后，必须立即推送到远程仓库对应分支。
- 工作分支之间不要互相合并，统一由 PM / 审核管理代理审核后合并到 `dev`。
- 合并到 `dev` 后，必须推送到 `origin/dev`，确保远端与本地保持同步。

## 八、上下文清理与协作状态恢复

- 当用户要求某个代理清理上下文、重置对话或开启新对话前，该代理必须先更新自己的角色状态文件。
- 角色状态文件至少记录：当前角色、当前工作树路径、当前分支、远端分支、正在进行的任务、已完成但未审核的分支、已派生子代理、需要用户确认的问题、最近一次提交或合并状态。
- 如果当前代理管理了子代理，必须先要求所有子代理各自更新自己的角色状态文件；确认子代理状态已记录后，当前代理再更新自己的角色状态文件。
- 清理上下文后的新对话中，代理必须先读取 `README.md`、`AGENTS.md`、自身角色说明文件和自身角色状态文件，再继续工作。
- 如果角色状态文件缺失、过期，或与 Git 实际状态冲突，必须先通过 `git status`、`git branch -a -vv`、`git worktree list`、`git log` 等命令核对，并向用户说明恢复结果。
- 主代理在重新开始工作后，必须先判断当前是否已经有对应的 subagent；如果没有，且当前任务涉及多个角色分工，就必须根据本文件的分工创建对应的 subagent，再继续处理任务。
- 主代理创建 subagent 时，必须先确认子 agent 是否已经存在；如果不存在，就先创建对应子 agent，并为其命名后再继续工作。
- 子 agent 在准备提交前，必须先同步 `dev` 的最新内容，再基于同步后的状态提交自己的改动，避免各自记录与 `dev` 差别过大。
- 子 agent 同步 `dev` 时，推荐按以下顺序执行：
  - 先 `git fetch origin dev`
  - 再将 `origin/dev` 合并或变基到当前工作分支
  - 如有冲突，先处理冲突并确认通过本地检查
  - 确认当前分支已包含最新 `dev` 后，再继续修改、提交和推送
- 在完成协作状态记录之前，不要主动清理上下文，也不要要求子代理清理上下文。
