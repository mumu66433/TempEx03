# 项目代理说明

## 一、全局原则

- 在进行任何修改之前，必须先阅读 [README.md](./README.md)，并将其作为本项目的首要规范。
- 如果任务要求与 `README.md` 冲突，默认以 `README.md` 为准，除非用户明确要求覆盖。
- 实现时请保持和仓库中描述的架构、启动流程、数据规则一致。
- 除非用户明确要求，否则不要在 `main` 分支直接开发、提交或推送。

## 二、分支与角色映射

- `dev`：PM / 审核管理代理。
- `design/planning`：策划设计代理。
- `design/ui`：UI 设计代理。
- `feature/program`：程序制作代理。

## 三、启动检查规则

- 每次开始工作前，必须检查当前 Git 分支。
- 根据当前分支确认自身角色，并读取 `docs/代理说明` 下对应的角色说明文件。
- 如果当前分支无法匹配任何角色，应先暂停并询问用户，不要直接修改文件。
- 如果当前任务与当前角色职责不一致，应先暂停并询问用户是否切换分支或调整任务。

## 四、角色说明文件

- PM / 审核管理代理：读取 [docs/代理说明/PM审核管理代理.md](./docs/代理说明/PM审核管理代理.md)。
- 策划设计代理：读取 [docs/代理说明/策划设计代理.md](./docs/代理说明/策划设计代理.md)。
- UI 设计代理：读取 [docs/代理说明/UI设计代理.md](./docs/代理说明/UI设计代理.md)。
- 程序制作代理：读取 [docs/代理说明/程序制作代理.md](./docs/代理说明/程序制作代理.md)。

## 五、目录权限边界

- 策划设计代理主要维护 `docs/策划案`，其他目录默认只读。
- UI 设计代理主要维护 `docs/原型制作`，其他目录默认只读。
- 程序制作代理主要维护 `client`、`server`、根目录工程脚本和工程配置；`docs/策划案` 与 `docs/原型制作` 默认只读。
- PM / 审核管理代理主要负责 `dev` 分支集成、审核记录、项目管理文档和分支协调。
- 非本角色维护目录默认只读；如需跨角色目录修改，必须先说明原因并征得用户同意。

## 六、提交与推送规范

- 每次完成修改后，必须提交代码，并使用中文提交信息说明本次修改内容。
- 每次提交后，必须将当前角色分支推送到远端对应分支。
- 工作分支之间不要互相合并，统一由 PM / 审核管理代理审核后合并到 `dev`。
- 合并到 `dev` 后，必须推送 `origin/dev`，确保远端与本地保持同步。
